# PuzzleMaster 谜题运行协议 v1

状态：`EXECUTABLE_PROTOTYPE`

对应实现：

- `src/runtime/contract.mjs`
- `src/runtime/registry.mjs`
- `src/runtime/breadth-first-solver.mjs`

## 1. 身份

每个谜题实例声明：

- `mechanismId`：原创具体机制，例如 `ORG-NAV-003`。
- `mechanismVersion`：规则语义版本。
- `runtimeKernel`：底层能力，例如 `TIME_EXPANDED_ROUTE`。

三者必须与机制注册表完全一致，否则运行时拒绝加载。

## 2. 状态

- 必须是JSON可序列化值。
- 由机制创建，不由渲染器创建。
- 状态转移必须返回新对象，禁止修改输入状态。
- 存档保存逻辑状态和动作历史，不保存画面对象、动画进度或像素坐标。

## 3. 动作

- 动作是显式对象，如 `{type: "WAIT"}`。
- 机制根据当前状态枚举合法动作。
- 非法动作返回 `{ok:false, reason}`，不能部分修改状态。
- 物理输入到逻辑动作的映射未来由表现层处理。

## 4. 状态转移

`applyAction(state, action, definition)` 必须：

- 确定性：相同输入始终产生相同输出。
- 纯函数式：不修改调用方状态和动作。
- 返回成功状态及可选逻辑事件，或明确失败原因。
- 不读取系统时间、帧率、DOM、Canvas或随机全局状态。

## 5. 目标与失败

每个机制独立实现：

- `isGoal(state)`
- `isFailure(state)`

运行时不会根据动画、按钮状态或文案猜测结果。

## 6. 哈希与求解

- `hashState` 必须包含所有影响未来合法动作和目标的状态。
- 等价状态必须产生相同哈希。
- 通用BFS输出最短长度、最短解集合、访问状态数和展开状态数。
- 专用求解器可以替换BFS，但报告格式必须兼容。

## 7. 注册表

机制注册时检查：

- ID一致
- 版本一致
- 运行时内核一致
- 必需方法完整
- ID不可重复注册

参考机制ID `REF-*` 只用于覆盖审计，不得注册为原创产品内容。产品机制统一使用 `ORG-*`。

## 8. 表现层边界

表现层未来负责：

- 把点击、拖动、滑动映射成动作
- 展示状态
- 播放逻辑事件对应动画
- 题面、提示、菜单和无障碍功能

表现层不得：

- 修改规则状态
- 根据碰撞动画自行判定合法性
- 用动画结束时间推进回合
- 保存渲染对象作为游戏存档

## 9. 当前合规检查

`npm run verify:runtime` 当前验证：

- 协议方法完整
- 定义与注册表一致
- 状态不被修改
- 转移确定性
- 状态可JSON往返
- 非法动作明确拒绝
- 重放结果与求解器一致
- 唯一最优解和状态数符合证据

后续每实现一个新内核，都必须加入至少一个合规样例。
