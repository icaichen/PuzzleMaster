# PuzzleMaster 交互适配器标准 v1

## 为什么需要适配器层

运行时内核只证明状态和规则可以复用，不证明玩家看到的画面和操作相同。例如 `GRID_PLACEMENT` 同时包含多格块铺装、行列计数、管道、道路面板和权重布局；这些机制可以共享网格、旋转、撤销和提交，但不能共享一套未经适配的题面。

系统分为：

```text
Runtime Kernel
  → Base Surface
    → Interaction Adapter
      → Puzzle Parameters
        → Theme / Art Assets
```

## 层级定义

- `Runtime Kernel`：状态、动作、转移、目标和求解。
- `Base Surface`：网格、图、自由画布、排序带、矩阵、立体展开等通用显示和输入能力。
- `Interaction Adapter`：机制特有的玩家动词、视觉语义、合法操作预览、错误反馈与提示定位。
- `Puzzle Parameters`：尺寸、对象、约束、目标和生成种子。
- `Theme / Art Assets`：不改变规则的最终美术表现。

## 允许共用同一适配器的条件

只有以下项目均一致时才能直接共用：

1. 玩家主要操作相同。
2. 可操作对象的选择与放置语义相同。
3. 非法操作反馈可以使用同一状态模型。
4. 提示能够指向同类视觉对象。
5. 移动端手势不会因机制变化而改变。

仅仅运行时内核相同，不构成适配器复用依据。

## 必须拆分适配器的条件

- 从点击选择变为拖动或画线。
- 从固定网格变为连续坐标。
- 需要时间轴、视角旋转、遮挡、光束或折叠等专属表达。
- 相同对象在另一机制中具有不同操作含义。
- 错误反馈和提示无法定位到相同视觉基元。

## 产品策略

- `BATCH_PROCEDURAL`：规则和视觉参数均可程序化，适合作为微信持续题库的主要来源。
- `CURATED_HYBRID`：逻辑可生成，但轮廓、图片或视觉歧义需要更严格审核，适合 App 精选题和微信低频投放。
- `BESPOKE_ONLY`：依赖专属叙事或视觉状态，不应承诺无人值守批量生产。

## 当前审计结论

96 种参考机制被映射为 28 类基础交互表面和 70 个实际机制适配器。这个数字是架构预算，不代表需要 70 个完全独立页面：基础控件和表面仍然复用；但每个适配器都需要自己的配置契约、反馈规则和至少一个自动试玩案例。

开发不能按 70 个适配器平均推进。应优先实现能覆盖多个机制、适合程序化量产、视觉依赖较低的适配器；高视觉和专属适配器留到基础审核流水线稳定以后。
