# NAV-03 温度交换路线机制

状态：`GENERATOR_PROTOTYPE`

## 核心关系

角色沿边移动时交换边两端的数值；边是否可通行又由这两个数值决定。因此移动同时改变角色位置和未来图结构。

温室与温度只是当前表现主题。机制层只处理节点、无向边、节点数值、允许差值、收集目标和终点。

## 状态

```text
position: 当前节点
values[node]: 每个节点当前数值
collected: 已取得目标的位集合
turn: 已执行动作数
```

## 动作

`MOVE(destination)`：目标必须与当前位置相邻，且两端当前数值之差等于 `requiredDifference`。

动作成功后：

1. 交换当前位置与目标节点的数值。
2. 角色移动到目标节点。
3. 若目标节点包含收集物，则加入 `collected`。
4. 回合数加一。

## 目标

在 `turnLimit` 内收集全部目标并抵达终点。

## 求解政策

- 主 App默认：唯一解或唯一最短解。
- 微信排行榜：唯一最短解，且步数上限等于最短解长度。
- 普通无尽模式可以接受少量等长最短解，但当前原型阶段仍强制唯一，便于验证生成质量。

## 生成参数

- 节点数
- 边数区间
- 数值范围
- 允许差值
- 收集物数量
- 目标最短解长度
- 最小决策点数量
- 最少动态解锁边数量
- 最大求解状态数

## 当前硬过滤器

1. 图必须连通。
2. 收集节点度数至少为2，避免纯粹进入死胡同再返回。
3. 最短解必须唯一且落在目标长度区间。
4. 标准解必须重复访问至少一个节点，证明状态变化参与规划。
5. 标准解必须使用至少一条初始不可通行、后来因交换而开启的边。
6. 标准解中至少两步存在多个合法选择，避免整题只是沿唯一通道前进。
7. 深度限制内状态数不能过低或失控。
8. 对保留起点、终点、收集点角色的图同构进行规范化去重。

## 当前软风险

- 数值交换若展示不清，会造成记忆负担而非推理乐趣。
- 生成器可能偏爱“访问支路后原路返回”的结构，需要额外多样性指标。
- 唯一最短解不等于唯一有趣解，也不等于难度准确。
- 正式批量生产前需加入提示生成和连续20关相似度窗口。
