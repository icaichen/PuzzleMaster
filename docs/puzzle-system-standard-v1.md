# PuzzleMaster 谜题产品、系统与质量标准 v1

状态：`DRAFT_FOR_PRODUCT_REVIEW`  
适用产品：PuzzleMaster 主 App、PuzzleMaster 微信小游戏  
本阶段范围：谜题内容与逻辑；不包含页面、美术、背景、棋子皮肤、动画或音效

## 1. 产品定义

PuzzleMaster 是一套共享谜题基础设施上的两个内容产品。

### 1.1 主 App：精品谜题库

- 谜题由 AI 逐题提出核心洞察、设计规则、构造实例、编写提示并实现验证。
- 允许专属机制和专属验证器，不要求所有题型可批量生成。
- 同一机制的续题必须增加新的推理关系，不能只扩大棋盘或替换数值。
- 机器验证负责证明规则一致、解空间符合要求；产品负责人负责最终体验验收。
- 目标是有限但高密度的题库：每道题都应提供不同或明显升级的思考体验。

### 1.2 微信小游戏：持续生成题库

- 只采用可参数化、可自动求解、可稳定分级的机制族。
- 每个机制族拥有自己的生成器和求解器，不建设“万能关卡生成器”。
- 允许玩家反复使用同一规则，但连续关卡必须通过结构相似度和难度变化过滤。
- 生成失败是正常结果；未通过硬门槛的候选关卡直接丢弃。
- 支持每日题、无尽模式、分段难度和可复现种子。

### 1.3 没有专业人工设计师时的责任边界

AI 负责：创意、规则、关卡、求解器、提示、批评、相似度检查、实现和修改。  
产品负责人负责：定义产品感觉、试玩验收、指出无聊/重复/困惑，并决定接受或淘汰。  

程序可以证明“答案条件”，不能单独证明“有趣”。因此任何自动分数都不能覆盖产品负责人的否决。

## 2. 三层系统，不混为模板

### 2.1 交互原语

交互原语只描述玩家如何表达动作：

`tap`、`select`、`drag`、`drop`、`rotate`、`slide`、`draw-line`、`draw-region`、`enter-value`、`sequence`、`undo`、`reset`、`submit`。

交互原语不包含谜题答案和推理规则。

### 2.2 机制族

机制族定义状态如何变化，例如“动作延迟两回合重放”或“上层非透明颜色遮盖下层”。机制族拥有独立的：

- 状态结构
- 合法动作枚举
- 状态转移
- 目标判定
- 状态哈希
- 求解器
- 可选生成器
- 难度特征提取器

### 2.3 谜题实例

谜题实例是可发布内容，包含特定布局、规则参数、题面、答案政策、提示和验证证据。主 App 的实例可以使用专属机制；微信实例必须引用可生成机制族。

## 3. 谜题生命周期

```text
IDEA
  → SPECIFIED
  → MACHINE_VERIFIED
  → CRITIC_REVIEWED
  → PLAYABLE
  → PRODUCT_ACCEPTED
  → RELEASED

任一步均可 → REJECTED
```

- `IDEA`：只有核心洞察，不算题目。
- `SPECIFIED`：规则、初始状态和目标无歧义。
- `MACHINE_VERIFIED`：求解器与规格检查全部通过。
- `CRITIC_REVIEWED`：完成公平性、原创性、捷径、重复度和提示审查。
- `PLAYABLE`：拥有最小可玩实现；美术不是前置条件。
- `PRODUCT_ACCEPTED`：产品负责人试玩接受。
- `RELEASED`：进入正式题库，内容不可静默修改，只能增加 revision。

## 4. 机制运行协议

每个机制必须实现等价于以下协议的能力；具体语言稍后确定：

```ts
interface PuzzleMechanism<State, Action, Parameters> {
  createInitialState(parameters: Parameters): State;
  enumerateActions(state: State): Action[];
  applyAction(state: State, action: Action): State | InvalidMove;
  isGoal(state: State): boolean;
  isFailure(state: State): boolean;
  hashState(state: State): string;
  solve(parameters: Parameters, limits: SolveLimits): SolveReport;
  explain(solution: Action[]): ExplanationStep[];
  generate?(seed: string, target: DifficultyTarget): Candidate[];
}
```

硬性边界：

- 状态与规则不得依赖画面对象。
- 相同初始状态和动作序列必须得到相同结果。
- 所有随机性来自显式 seed。
- 规则层不得读取动画时长、像素位置或设备帧率。
- 存档保存逻辑状态和动作历史，不保存渲染器对象。
- 表现层只能展示状态，不能成为答案判定来源。

## 5. 谜题定义契约

机器可读定义见 [`puzzle-definition.schema.json`](../schemas/puzzle-definition.schema.json)。主要字段：

- `identity`：稳定 ID、revision、语言无关机制标识。
- `products`：允许进入主 App、微信端或两者。
- `taxonomy`：认知类型、交互类型、表现结构。
- `content`：题面、玩家可见规则、分层提示和反馈文案。
- `runtime`：机制键、初始状态、参数、动作定义。
- `solutionPolicy`：唯一解、多解、最短解等发布政策。
- `difficulty`：目标级别与机器测量特征。
- `verification`：求解器版本、解数量、最短长度和可复现证据。
- `originality`：去皮后的结构指纹、比较对象和风险状态。

题面与答案不得成为唯一规则来源。验证器必须能够只读取结构化 `runtime` 数据完成判定。

## 6. 解政策

不是所有高质量谜题都必须唯一解，但发布前必须明确选择一种政策：

- `UNIQUE`：只允许一个完整解。
- `UNIQUE_OPTIMAL`：可能有多个可行解，但最优解唯一。
- `FINITE_ACCEPTED`：允许有限个等价答案，必须全部记录或由规范化规则覆盖。
- `OPEN_CONSTRAINT`：任何满足约束的构造均接受；仅适用于创造型题目，不进入依赖唯一答案的排行榜。

默认政策：主 App 使用 `UNIQUE` 或 `UNIQUE_OPTIMAL`；微信排行榜题必须使用 `UNIQUE`、`UNIQUE_OPTIMAL` 或具有稳定最优分数的 `FINITE_ACCEPTED`。

## 7. 难度模型

难度不能只用最短步数表示。每题记录以下证据：

- 初始合法动作数量和平均分支因子
- 可达状态数
- 最短解长度
- 最短解数量
- 错误动作后仍可恢复的比例
- 必须同时追踪的状态变量数量
- 规则交互层数
- 关键洞察出现前的合理误导数量
- 提示前后搜索空间缩减比例

产品难度采用五级：

| 等级 | 预期体验 | 设计要求 |
|---|---|---|
| 1 入门 | 1–2分钟理解并完成 | 单一规则关系，几乎无惩罚 |
| 2 轻度 | 3–5分钟 | 一次局部推理或短规划 |
| 3 标准 | 5–10分钟 | 两条规则组合，存在合理错误路线 |
| 4 困难 | 10–20分钟 | 多阶段推理，需要记录中间状态 |
| 5 挑战 | 20分钟以上 | 多层联动，但不能依赖盲目穷举 |

机器只给出建议等级；在没有大规模玩家数据前，不宣称机器评级等于真实难度。

## 8. 提示标准

每道主 App 谜题至少三层提示：

1. **观察提示**：指出应关注的关系，不提供具体动作。
2. **洞察提示**：解释核心机制如何影响解题。
3. **推进提示**：给出局部起步或中间结论，仍保留最后推理。

可选最终提示可以接近解答，但必须明确标注。微信生成题的提示优先由求解路径和状态差异自动生成，并通过一致性检查，禁止使用无法由验证器证明的自然语言建议。

## 9. 原创性标准

原创性比较的是“裸机制和解法结构”，不是题材名称。

每题保存结构指纹：

- 状态变量及其域
- 玩家动作集合
- 状态转移关系
- 目标条件
- 关键不变量
- 预期核心洞察
- 标准解的因果结构

以下任一情况默认淘汰：

- 只替换角色、物体或背景。
- 布局做平移、旋转、镜像后与已知题相同。
- 数值变化但推导步骤完全相同。
- 提示顺序可一一对应参考题。
- 去掉美术后，普通玩家会直接认出是同一题。

“通用机制已存在”不自动构成淘汰；但实例必须在约束组合、推理洞察或交互反馈上形成实质变化。所有相似性结论都记录为风险判断，不作法律结论。

## 10. 双产品发布门槛

### 10.1 所有谜题的硬门槛

- 数据契约有效，ID 和 revision 合法。
- 规则确定、可复现，不依赖渲染状态。
- 解政策得到求解器证明。
- 没有未声明的死局、越界或无响应动作。
- 提示与实际解一致。
- 每条玩家可见规则至少影响一个合法性或目标判断。
- 通过结构去重和镜像/旋转归一化检查。
- 不包含受保护角色、题面、布局、名称、图像或解答复刻。

### 10.2 主 App 软评分（100分）

| 维度 | 分值 |
|---|---:|
| 核心洞察的新鲜度 | 20 |
| 解开时的满足感 | 15 |
| 公平性与表达清晰度 | 15 |
| 推理结构的优雅程度 | 15 |
| 与现有题库的差异 | 15 |
| 交互是否参与推理 | 10 |
| 提示质量 | 5 |
| 视觉化潜力 | 5 |

进入可玩原型门槛：机器与批评流程建议分不低于 80。  
进入正式 App 门槛：建议分不低于 85，且产品负责人明确接受。  
产品负责人可以否决高分题；低分题不能仅靠主观喜欢绕过硬门槛。

### 10.3 微信生成题门槛

- 生成 seed 可复现。
- 求解在规定资源内完成。
- 符合目标难度区间。
- 与最近 20 关的结构相似度低于机制自定阈值。
- 单局目标时长通常为 30秒–5分钟。
- 自动提示能够从求解路径生成。
- 机制生成器离线批量接受率达到约定值；接受率过低意味着生成器设计需要修改，而不是放宽质量。

## 11. AI 生产与批评流程

每道精品题由不同职责的检查阶段依次处理：

1. **设计阶段**：提出核心洞察和最小规则。
2. **形式化阶段**：转为状态、动作、转移和目标。
3. **构造阶段**：从目标体验反向构造实例。
4. **求解阶段**：穷举、搜索或约束求解，生成完整解报告。
5. **对抗阶段**：主动寻找捷径、歧义、规则漏洞和非预期解。
6. **原创性阶段**：与内部题库和参考目录比较结构指纹。
7. **提示阶段**：从求解因果链生成三级提示，并逐条验证。
8. **产品阶段**：制作无美术原型供产品负责人试玩。
9. **决策阶段**：接受、修改或淘汰。

微信机制在第 3 步替换为参数化生成，随后对每个候选重复第 4–7 步。

## 12. 题库组合标准

主 App 不能因单题合格而形成整体重复。题库按以下维度控制分布：

- 认知类型
- 输入方式
- 平面/序列/网络/数值等状态表示
- 是否实时变化
- 是否允许撤销
- 解题时主要依赖观察、演绎、规划或构造
- 题目时长与难度
- 机制首次出现或递进变体

首发规划在机制蓝图确定后冻结。任何新题加入时都重新计算题库分布，而不是只审查单题。

## 12.1 参考覆盖基线

系统广度不再由预估机制数量证明，而由逐题覆盖矩阵证明。当前参考基线为：

- 参考题：165题，规则文本覆盖165/165。
- 具体机制模式：96个；系列续题归入同一模式。
- 运行时能力内核：30个。
- 每道参考题必须且只能映射到一个主要机制模式，并可附加次要能力。
- 新发现的参考规则若无法由现有内核表达，必须先更新覆盖矩阵和内核蓝图，不能在单题代码里偷偷硬编码。
- 100%覆盖当前指“详细题面与提示层面的结构分类”；依赖原始图像才能确认的几何细节继续标记为视觉复核项。

## 13. 版本与修改规则

- `schemaVersion`：数据格式版本。
- `mechanismVersion`：规则语义版本。
- `revision`：谜题实例修订号。
- 已发布谜题若规则或答案改变，必须增加 revision 并重新运行全部验证。
- 纯文案错字可以增加 patch revision，但仍需题面一致性检查。
- 求解器升级后，所有依赖该机制的已发布题必须重新验证。

## 14. 当前阶段完成定义

在进入前端和美术前，必须完成：

- 本标准由产品负责人确认。
- 165题覆盖矩阵保持100%，且没有重复归类。
- 30个运行时内核均有明确状态、动作、验证与扩展边界。
- JSON Schema 和至少三个不同机制的实例通过验证。
- 至少一个精品专属机制、一个批量生成机制完成完整逻辑原型。
- 自动报告能输出解数量、最短解、难度特征、结构指纹和提示一致性结果。
